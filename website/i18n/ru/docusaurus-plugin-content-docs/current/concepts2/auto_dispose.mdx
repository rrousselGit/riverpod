В Riverpod можно настроить автоматическое освобождение ресурсов, 
связанных с provider, когда он больше не используется.

## Включение и отключение автоматического освобождения ресурсов

Если вы используете генерацию кода, эта возможность включена по умолчанию. 
Её можно отключить через аннотацию:
```dart
// Отключить автоматическое освобождение ресурсов
@Riverpod(keepAlive: true)
String helloWorld(Ref ref) => 'Hello world!';
```

Если генерация кода не используется, автоматическое освобождение ресурсов можно включить с помощью параметра `isAutoDispose: true`
при создании provider:
```dart
final helloWorldProvider = Provider<String>(
  // Включить автоматическое освобождение ресурсов
  isAutoDispose: true,
  (ref) => 'Hello world!',
);
```

:::note
Включение или отключение автоматического освобождения ресурсов не влияет на то, 
будет ли состояние уничтожено при пересчёте provider.  
При пересчёте provider его состояние всегда уничтожается.
:::

:::caution
Если provider принимает параметры, рекомендуется включать автоматическое освобождение ресурсов.
В противном случае для каждой комбинации параметров будет создаваться отдельное состояние, 
что может привести к утечкам памяти.
:::

## Когда срабатывает автоматическое освобождение ресурсов?

Когда автоматическое освобождение ресурсов включено, Riverpod отслеживает, есть ли у provider слушатели.
Для этого учитываются вызовы [Ref.watch](https://pub.dev/documentation/hooks_riverpod/latest/hooks_riverpod/Ref/watch.html), [Ref.listen](https://pub.dev/documentation/hooks_riverpod/latest/hooks_riverpod/Ref/listen.html)
и некоторых других методов.  

Когда количество слушателей становится равным нулю, provider считается "неиспользуемым", и вызывается
[Ref.onCancel](https://pub.dev/documentation/hooks_riverpod/latest/hooks_riverpod/Ref/onCancel.html).
После этого Riverpod ожидает один кадр (ср. await null). Если по прошествии этого кадра 
provider всё ещё не используется, он уничтожается, после чего вызывается [Ref.onDispose](https://pub.dev/documentation/hooks_riverpod/latest/hooks_riverpod/Ref/onDispose.html). 

## Реакция на уничтожение состояния

В Riverpod существует несколько встроенных сценариев, при которых состояние может быть уничтожено:

- provider больше не используется и работает в режиме автоматического освобождения ресурсов (подробнее об этом позже).
  В этом случае всё состояние, связанное с provider, уничтожается.
- provider пересчитывается, например из-за `ref.watch`.
  В этом случае предыдущее состояние уничтожается, а вместо него создаётся новое.

В обоих случаях может потребоваться выполнить какую-либо логику при уничтожении состояния.  
Для этого существует `ref.onDispose`. Этот метод позволяет 
зарегистрировать обработчик, который будет вызываться каждый раз при уничтожении состояния.

Например, его можно использовать для закрытия активного `StreamController`:

<AutoSnippet {...onDisposeExample} />

:::caution
Колбэк, передаваемый в `ref.onDispose` не должен вызывать побочные эффекты.
Изменение состояния providers внутри `onDispose` может привести к непредсказуемому поведению.
:::

:::info
Существуют и другие полезные события жизненного цикла:

- `ref.onCancel` — вызывается, когда у provider удаляется последний слушатель.
- `ref.onResume` — вызывается, когда после вызова `onCancel` к provider добавляется новый слушатель.

:::

:::info
Вы можете вызывать `ref.onDispose` столько раз, сколько потребуется.
Не стесняйтесь вызывать его отдельно для каждого объекта, требующего освобождения ресурсов, внутри provider. Такой подход 
упрощает поиск ситуаций, когда какой-либо объект был забыт и не был корректно освобождён.
:::

## Принудительное уничтожение provider с помощью `ref.invalidate`

Иногда может потребоваться принудительно уничтожить provider.
Это можно сделать с помощью `ref.invalidate`, который можно вызывать как из другого provider, так и из виджета.

При вызове `ref.invalidate` текущее состояние provider уничтожается.
После этого возможны два варианта:

- если provider отслеживается, для него будет создано новое состояние.
- если provider не отслеживается, он будет полностью уничтожен.

<AutoSnippet raw={invalidateExample} />

:::info
Провайдеры также могут инвалидировать сами себя с помощью `ref.invalidateSelf`.
Однако в этом случае всегда будет создаваться новое состояние.
:::

:::tip
Если вы хотите инвалидировать provider, принимающий параметры, 
можно инвалидировать только одну конкретную комбинацию параметров
или сразу все комбинации параметров:

:::

## Тонкая настройка освобождения ресурсов с `ref.keepAlive`

Как уже упоминалось выше, при включённом автоматическом освобождении ресурсов состояние уничтожается, 
если у provider нет слушателей в течение одного полного кадра.

Однако иногда требуется более точный контроль над этим поведением. 
Например, вы можете захотеть сохранять результат успешных сетевых запросов, 
но не кэшировать неудачные.

Это можно реализовать с помощью `ref.keepAlive`, после включения автоматического освобождения.
С его помощью вы сами решаете _когда_ состояние перестаёт автоматически уничтожаться.

<AutoSnippet {...keepAlive} />

:::note
Если provider пересчитывается, автоматическое освобождение ресурсов снова включается.

Также можно использовать возвращаемое значение `ref.keepAlive` для возврата к автоматическому освобождению ресурсов.
:::

## Пример: сохранение состояния на заданное время

В настоящее время Riverpod не предоставляет встроенного механизма для сохранения состояния 
на строго заданный промежуток времени.  
Однако реализовать такую логику легко и удобно с использованием уже рассмотренных инструментов.

С помощью `Timer` + `ref.keepAlive` можно удерживать состояние provider активным в течение определённого времени.

если provider не использовался (не имел слушателей) в течение определённого промежутка времени.
